나도 근본을 지향하는 사람입니다

이 글 보고 제가 ORM 찬양론자인 줄 알았다면 오해입니다.

SQL이 얼마나 강력한 도구인지 알고, 직접 작성한 SQL이 퍼포먼스에서 우월하다는 것도 압니다.

ORM이 종종 이상한 쿼리 생성해서 삽질하는 경우도 알고 있습니다

그렇다고 해서 ORM을 무조건 배척하는 것도 이상한 일입니다.

ORM이 필요한 상황이 있고 직접 SQL을 짜야 하는 상황도 있습니다.

근본만 따지다가 유지보수하다가 멘탈 터질 수도 있으니 적당히 현실적으로 접근하는 게 좋습니다.

SQL만 신봉하다가 프로젝트 터지면

"이게 다 ORM을 안 써서 그런 거다"라고 할 수도 없습니다. 그냥 본인 선택임

SQL을 꼭 직접 짜야 합니까

SQL 근본주의자들이 자주 하는 말이 있습니다

  • ORM 쓰면 쿼리 최적화가 안 됨
  • SQL 직접 작성하는 게 정석임
  • ORM 쓰면 쿼리가 어떻게 나가는지도 모름

다 맞는 말이긴 한데 현실에서는 이거보다 중요한 게 많습니다.

개발 속도, 유지보수 편의성, 협업의 효율성이 더 중요할 때가 있습니다.

ORM을 적절히 활용하면 코드가 깔끔해지고 생산성이 올라갑니다.

SQL 직접 다 짜고 있으면 그냥 SQL 개발자가 되는 겁니다.

이러다 보면 SQL만 치다가 회사에서 DBA로 전직하게 될 수도 있음

ORM이 필요한 경우

  1. CRUD 위주의 서비스
    • 단순한 데이터 저장, 조회 작업이 많으면 ORM이 답입니다
    • SQL 직접 짜는 게 의미 없는 노동이 될 가능성이 큽니다. 그럴꺼면 AI가 님들보다 잘짬
    • 모든 CRUD를 직접 SQL로 작성하면 팀원들한테 "이걸 왜 이렇게 힘들게 짜셨죠?"라는 질문을 받을 수도 있음
  2. 비즈니스 로직이 복잡할 때
    • 테이블 여러 개 묶어서 쿼리 날릴 일이 많다면 ORM이 유지보수를 훨씬 쉽게 만들어줍니다
    • SQL 직접 짜다 보면 코드가 길어지고 복잡해지는 경우가 많습니다
    • 예전에 SQL 100줄짜리 SELECT 문 보면서 멘탈 터진 경험이 있다면 ORM이 왜 필요한지 알 겁니다
  3. 여러 명이 협업하는 프로젝트
    • SQL 스타일 다 다르면 코드 리뷰할 때 불필요한 논쟁이 생깁니다
    • ORM을 사용하면 기본적인 코드 스타일이 통일됩니다
    • 팀원마다 SQL 스타일 다르면 "이거 PostgreSQL에서 돌아가나요?" 같은 얘기가 나오기 시작함
  4. 테이블이 자주 변경되는 경우
    • SQL 직접 다루면 테이블이 바뀔 때마다 쿼리 수정하는 게 일이 됩니다
    • ORM을 사용하면 모델만 수정하면 끝입니다
    • 테이블 바뀔 때마다 모든 SQL을 다시 작성하는 거, 해본 사람은 압니다. 멘탈 무너짐

이걸 무시하면 테이블 하나 바뀔 때마다 쿼리 전부 뜯어고치는 자신을 발견하게 될 수도 있습니다

ORM을 쓰면 안 되는 경우

  1. 대용량 트래픽이 핵심인 서비스
    • ORM이 자동 최적화한다고 해도 직접 튜닝한 SQL보다 성능이 떨어질 때가 많습니다
    • 트래픽 터지면 CTO한테 불려가서 왜 직접 SQL 안 짰냐는 소리 들을 수 있습니다
    • "이거 ORM이 이상한 쿼리 날려서 그런 겁니다"라고 말하는 순간 CTO의 얼굴이 굳어질 수도 있음
  2. 복잡한 SQL 기능을 자주 사용해야 할 때
    • 윈도우 함수, 서브쿼리, DBMS별 최적화 기능 등을 써야 하면 ORM이 걸림돌이 될 수도 있습니다
    • ORM이 알아서 해줄 거라고 기대하지 않는 게 좋습니다
    • ORM 믿고 있다가 "이건 ORM으로 못 합니다"라고 하면 결국 직접 SQL 짜야 하는 상황이 옴
  3. DB 성능이 최우선일 때
    • 데이터베이스 퍼포먼스가 서비스 핵심이면 직접 SQL을 작성하는 게 맞습니다
    • DBA 앞에서 ORM 얘기 꺼냈다가 한숨 듣기 싫으면 조용히 SQL 짜는 게 답입니다
    • DBA한테 "ORM 써도 되죠?"라고 물어보는 순간, 이미 마음속으로 ORM은 퇴출됨

결론

ORM이 무조건 답은 아닙니다.

하지만 필요할 때는 써야 합니다.

ORM 없으면 CRUD 짜는 데 시간 다 씁니다.

성능이 중요한 곳에서는 직접 SQL을 짜는 게 맞습니다.

결국 개발은 정석이 아니라 상황에 맞는 선택을 하는 게 중요합니다

ORM을 무조건 배척하는 사람들, 나중에 테이블 구조 바뀔 때 직접 SQL 수정하면서 후회할 수도 있습니다.

그때라도 늦지 않았으니 ORM을 받아들이면 됩니다.

"ORM 절대 안 씀" 같은 선언적 개발 철학은 현실에서 오래 못 갑니다. 적당히 유연하게 가야 합니다
