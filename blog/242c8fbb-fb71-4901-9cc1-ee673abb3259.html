  1. 켄트 벡의 Tidy First?
  • 코드 정리 ≠ 리팩토링
    • 모양과 행위는 같으나, 목적이 다름
  • 코드 정리, 리팩토링, 설계 등 코드를 작성/수정하는 모든 일은 가장 작은 단위부터 실행해야 한다. (Bottom Up)
  • 코드 정리
    • 내가 모르는 코드여도 내가 코드를 쉽게 건들 수 있도록 변화시키는 것이 중요하다.
    • 코드 정리는 검토의 범위에 포함되지 않아야 한다.
    • 코드 정리는 업무 및 기존 기능에 영향이 가면 안된다.
  • 설계
    • 요소를 식별하여, 유익하게 관계맺는 일
    • 미래를 위해 유기체적처럼 구성하는 과정

  1. 왜 나는 테스트를 작성하기 싫을까?
  • 테스트를 작성, 유지보수하는 비용이 테스트 후 얻는 이득보다 크다.
  • 제거할 수 있는 테스트 비용
    • Given - 테스트를 복잡하게 작성하려 한다. - 작성 비용
    • Then - 하나의 테스트에서 많은 검증을 하려 한다. - 유지보수 비용
  • 테스트 비용 줄이기
    • Given - 가장 간단한 case에서 테스트
    • Then - 가장 적게 검증하기
  1. 실전! MSA 개발 가이드
  • 서비스간 통신은 가능, 각 서비스가 사용하는 DB를 서로 공유하면 안됨
  • N+1 문제, DDoS를 피하기 위해 일괄 조회
  • 서비스별 로컬 캐시 이용 (동기화 X)
  • 성능 비교: 일괄 조회 & Cache > SQL JOIN (모놀리식) > 일괄 조회 > N+1
  • 트랜잭션 보장성
    • 원자성
      • A 서비스 create → B 서비스 create → A 서비스 update 중 에러가 발생하여 Rollback, 이 때 B 서비스는 Rollback되지 않아 A 서비스의 DB와 정합성이 맞지 않음
        • 서비스 간 통신하는 작업에 대해서는 중간에 이벤트로 분리
        • 모델링 변경 (A ∩ B 테이블을 연관관계 주인인 서비스 DB에 배치)
        • 구현이 너무 힘들면 서비스 병합하여 경계를 없애자
    • 독립성
      • A 서비스 작업 후 바로 B 서비스 작업 시 애플리케이션에서 A 서비스와 B 서비스 간의 데이터가 일시적으로 안맞을 수 있음
  • Conclusion
    • 패턴
      • 꼭 필요한 속성만 복사
      • 일괄 조회
      • 로컬 캐시
      • 전체 취소 안 하면 이벤트(RabbitMQ, Kafka 등)로 쓰기
      • 멱등성
      • 분산 모델
      • 데이터는 오너쉽 가진 서비스에
    • 안티 패턴
      • 테이블 스키마 복사
      • 병렬 호출
      • API 재시도
      • 로컬 캐시 동기화
