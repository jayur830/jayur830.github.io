[Next.js] getStaticProps, getStaticPaths 설명

2024-10-30

Next.js는 기본적으로 SSR이다.

정적인 UI들은 빌드 시 모두 html 파일로 생성하여

처음 URL 접속 시 미리 pre-rendering 된 html을 그대로 보여준다.

하지만, 서버 통신에 의한 데이터 fetching이 필요한 경우

클라이언트에서 비동기 통신으로 가져올 수 밖에 없다.

만약 이 데이터가 극단적으로 생각해서 1억건, 혹은 그 이상의 개수일 경우,

페이지 로딩 속도는 그 놈의 데이터 하나 가져오는거 때문에

굉장히 느려지게 된다.

여기서 우리는 한 가지 상상을 할 수 있다.

pre-rendering 하는 김에 데이터도 fetch 해서 같이 pre-render 하면 어떨까?

Next.js에서는 이것을 상상이 아닌 현실로 옮길 수 있다.

Build Time에 Data Fetching까지 모두 Pre-render 하여

완전한 html을 생성하여 클라이언트에 별도의 서버처리 없이 그대로 보여주는 것을

**SSG(Static Site Generation)**라고 한다.

그리고 SSG를 적용하기 위해 필요한 함수가 두 가지가 있는데,

이 글의 제목과 같이 getStaticProps, getStaticPaths이다.

getStaticProps

데이터 Fetching을 서버단에서 미리 하여 Pre-render 하도록 하는 함수이다.

예시로 다음과 같은 코드가 있다.

// pages/home/index.page.tsx

import { NextPage } from 'next';

interface Props {
  data: any[];
}

const Home: NextPage<Props> = ({ data }) => {
  return (
    <div>
      {data.map((obj) => { ... })}
    </div>
  );
};

export default Home;

export async function getStaticProps() {
  const data = await fetch(url).then((response) => response.json());
  
  return {
    props: {
      data,
    },
  };
}

사용법은 위와 같이 페이지 컴포넌트 바깥의 전역에

getStaticProps 함수를 export 하면 Next.js가 이를 알아서 인식한다.

export 하는 getStaticProps는 해당 페이지에 있어야 그 페이지로 적용된다.

그리고 axios나 fetch 등의 비동기 통신을 통해 데이터를 받아와

props의 프로퍼티로 리턴하게 되면,

해당 페이지 컴포넌트의 props로 그대로 들어오게 된다.

주의사항)

서버단에서만 실행되어 브라우저 관련 코드는 적용되지 않는다. (예 - window, document, React Hooks 등)

getStaticPaths

위에서 설명한 getStaticProps는 Pre-render에 필요한 데이터들을

서버단에서 미리 처리하여 props로 내려주는 역할이었다면,

이 함수는 동적 라우팅의 가능한 모든 경우의 수의 html을 전부 생성하도록 하는 함수이다.

예를 들면, /insight/[category]/[id] 라는 path가 있고,

getStaticPaths 함수를 정의한다면,

(category의 개수 × id의 개수) 만큼의 html 파일이 Build Time 때 미리 생성된다.

바로 코드로 보자고.

// pages/insight/[category]/[id]/index.page.tsx

import { NextPage } from 'next';

const Insight: NextPage = () => {
  ...
  
  return (...);
};

export default Insight;

export async function getStaticPaths() {
  const menus = await fetch(url).then((response) => response.json());
  
  const paths = menus.map(({ id, category }: any) => ({
    params: { id, category },
  }));
  
  return {
    paths,
    fallback: true,
  };
}

getStaticProps와 마찬가지로 async 함수를 export 하는 형태로 정의한다.

리턴 값은 paths, fallback인데, paths에 가능한 모든 라우팅 경우의 수의 객체가 포함되고,

fallback은 paths에 없는 파라미터로 URL 접속했을 경우에 대해

404를 반환할지(false), 아니면 새로 html을 만들어 보여줄지(true)의 여부를 결정한다.

주의사항)

서버단에서만 실행되어 브라우저 관련 코드는 적용되지 않는다. (예 - window, document, React Hooks 등)

2023.02.26 추가)

간만에 Next.js를 했더니 뭔 13버전으로 또 업데이트했더라고.

다른건 없고 getStaticProps, getStaticPaths에 대해서 추가된 내용이 있어 그거 좀 적어보려 한다.

export async function getStaticProps() {
  const data = await fetch(url).then((response) => response.json());
  
  return {
    props: {
      data,
    },
    revalidate: 10,
  };
}

revalidate라는 놈이 생겼다.

저기 적힌 숫자가 초(second)인데, 빌드 후 저 getStaticProps가 있는 페이지를 실행하면

일단 한번 실행되고, 저 시간만큼 있다가 Next 서버 백그라운드에서 또 한번 실행된다.

중요한 건 개개인마다 요청한 시점이랑 상관없이 그냥 서버단 기준으로 revalidate(초)만큼 있다가 실행된다.

이게 스케줄링이 도는 듯

export async function getStaticPaths() {
  const menus = await fetch(url).then((response) => response.json());
  
  const paths = menus.map(({ id, category }: any) => ({
    params: { id, category },
  }));
  
  return {
    paths,
    fallback: 'blocking',
  };
}

getStaticPaths의 fallback에 'blocking'이라는 놈이 새로 생겼다.

이제 이렇게 하면 빌드 타임 때 생성했던 path들 외에

새로운 path의 param들이 들어오면 그거에 대해서 Next.js 서버 내에서

새로 또 html을 만든다.

Next.js 13 버전에는 이거 말고도 여러가지가 더 나왔는데 자세한 건 따로 다루도록 한다.
