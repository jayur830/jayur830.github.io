프레임워크는 종교가 아닙니다

2025-03-22

프레임워크는 가이드일 뿐, 법이 아님

개발하다 보면 이런 고민이 한 번쯤은 듭니다

'이 프레임워크의 방식대로 다 맞춰야 하나, 아니면 적당히 벗어나도 되나?'

정답부터 말하면 틀을 따르는 게 맞지만, 종속되면 안 됩니다

어떤 프레임워크나 라이브러리를 도입했다면 그 철학과 방식에 맞춰서 개발하는 게 기본입니다

그게 유지보수도 쉽고 협업할 때도 좋습니다

하지만 모든 걸 거기에 맞출 필요는 없습니다

프레임워크는 도구일 뿐입니다

우리를 편하게 해주려고 있는 거지 우리를 구속하려고 있는 게 아닙니다

프레임워크가 ‘이거 꼭 이렇게 써야 돼’라고 말해도, 가끔은 “싫은데요?”라고 할 줄도 알아야 합니다

물론 우리는 프레임워크를 숭배할 수도 있습니다

하지만 그것이 곧 프레임워크의 노예가 되는 길임을 잊지 마십시오

AI도 문서 보고 코드를 짜는 시대입니다

사람까지 문서에 얽매이면 나중에는 “AI가 대신 개발하는 게 낫겠네요” 소리 듣습니다

프레임워크를 따르면 좋은 점

✅ 고민할 필요 없음

예를 들어 Next.js의 App Router를 쓰면 라우팅, 데이터 패칭 방식 같은 게 이미 정해져 있습니다

NestJS를 쓰면 컨트롤러, 서비스, 모듈 구조가 정해져 있고요

이걸 그대로 따르면 처음부터 고민할 필요 없이 빠르게 개발할 수 있습니다

(물론 공식 문서 읽는 것은 필수입니다. 아니면 프레임워크한테 맞고 시작하세요)

✅ 공식 문서가 해결책을 줌

틀 안에서 개발하면 공식 문서나 커뮤니티에서 이미 검증된 해결책을 찾을 수 있습니다

프레임워크 방식대로 짜면 남들도 비슷한 문제를 겪었고 답도 많습니다

이런 거 찾아볼 때 공식 문서 안 보고 스택오버플로 먼저 뒤지는 분들이 있는데

그러면 “이거 쓰세요” “아니요 저거 쓰세요” 같은 답변만 잔뜩 보고 현타 옵니다

✅ 협업할 때 익숙한 구조로 진행 가능

팀원들도 같은 프레임워크를 쓰니까 코드 구조가 익숙합니다

사람마다 방식이 다르면 코드 읽기가 어려운데 정해진 틀 안에서 작업하면 누구나 이해하기 쉽습니다

혼자 개발하는 게 아니라면 어느 정도는 프레임워크의 가이드를 따르세요

그렇지 않으면 팀원들이 당신을 욕할 것입니다 (진짜임)

틀을 벗어나야 하는 경우

하지만 모든 걸 프레임워크 방식대로 해야 하는 것은 아닙니다

때로는 틀을 벗어나야 할 순간이 옵니다

✅ 프레임워크 방식이 비효율적일 때

예를 들어 Next.js의 fetch + useEffect 조합은 간단해 보이지만,

모든 데이터를 클라이언트에서 패칭하면 TTFB가 길어지고 성능이 나빠질 수도 있습니다

이럴 때는 fetch를 서버 컴포넌트에서 실행하고, 필요한 데이터만 클라이언트로 내려주는 방식이 더 효율적일 수 있습니다

또한 Next.js의 Server Actions 같은 기능이 있지만, 모든 비동기 처리를 이걸로 하면 오히려 코드 복잡도가 올라갈 수도 있습니다

이럴 땐 그냥 기존의 API 라우트를 활용하는 것이 더 낫습니다

프레임워크가 시키는 대로 하면 “Best Practice”라고 하지만

가끔은 “이거 Best 맞습니까?” 싶을 때도 있습니다

님들 애기 아니니깐 그럴 땐 직접 테스트해보고 결정하세요

✅ 프로젝트 요구사항과 안 맞을 때

모든 프레임워크가 100% 완벽한 것은 아닙니다

React로 개발한다고 해서 모든 상태를 React 방식으로 관리할 필요는 없습니다

Redux를 쓰고 있지만 어떤 상태는 그냥 useState로 관리하는 것이 더 깔끔할 수도 있습니다

개발하다 보면 프레임워크가 예상하지 못한 상황이 나오기도 합니다

이럴 때는 ‘공식 문서에 없는데?’ 하면서 당황하지 말고 적당히 해결하세요

(물론 해결한 다음에 팀원들한테 “이게 공식 문서에 없긴 한데...” 라고 말하면 욕 먹을 가능성이 있습니다)

✅ 확장성을 고려할 때

지금은 특정 프레임워크를 쓰지만 나중에 바꿀 수도 있습니다

그런데 모든 코드를 특정 프레임워크 방식으로만 짜면 바꾸기가 어려워집니다

핵심 로직은 독립적으로 관리하고 프레임워크 의존성은 최소화하십시오

이건 마치 집을 지을 때 가구를 전부 벽에 못 박아서 고정하는 것과 같습니다

처음엔 튼튼하고 좋아 보이는데 나중에 이사 갈 때 다 뜯어내야 한다는 걸 깨닫게 됩니다

프레임워크 방식만 따라가다 보면 나중에 “어? 이거 바꿀 수가 없는데?”라는 상황이 옵니다

결론: 도구는 편하게 쓰되, 주인은 내가 돼야 함

프레임워크를 쓴다면 기본 틀을 따르는 것이 맞습니다

그게 더 빠르고 유지보수도 쉽습니다

하지만 틀에 갇히면 안 됩니다

프레임워크는 우리를 돕기 위한 도구일 뿐이지 종교가 아닙니다

가끔은 공식 문서에도 없는 길을 걸어야 할 때가 있습니다

필요할 때는 과감하게 틀을 벗어나세요

언제나 프레임워크가 주인이 아니라 내가 주인이어야 합니다

프레임워크를 숭배하지 마십시오

프레임워크가 여러분을 구원해주지는 않습니다

AI가 프레임워크 문서 읽고 코드를 짜는 시대입니다

사람까지 문서만 보고 기계처럼 개발하면 그냥 AI한테 대체당하는 게 더 빠릅니다

도구를 쓸 줄 아는 사람이 도구에 종속되면 그건 그냥 프레임워크의 시다바리가 되는 것입니다

그러지 말고 필요할 때는 프레임워크한테 “이건 내가 결정하겠습니다” 라고 말할 수 있어야 합니다
