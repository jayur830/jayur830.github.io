개발자는 부지런하면서도 때론 게을러야 한다.

반복적으로 일어나는 작업에 대해서는

자동화를 통해 업무 효율을 높여야 한다.

Jenkins, CodeDeploy 등과 같은 CI/CD 툴로 보통 배포 자동화를 많이들 하는데,

Github에서 제공하는 Github Action으로도 가능하다고 한다.

그래서 해버렸다. 실제 서비스 돌아가고 있다.

어떻게 하냐면,

일단 내 프로젝트가 Git에 연동 되어있어야 한다. (Github Action으로 하니 당연한 말이다.)

프로젝트 루트 디렉토리에서 .github 라는 디렉토리를 생성하고

그 안에 workflows 라는 디렉토리를 생성하자.

그리고 workflows 디렉토리안에 yml 파일 하나를 작성한다. 파일명은 아무거나 해도 된다.

난 gradle-publish.yml 라는 파일명으로 했다.

이 yml에는 아래와 같은 걸 작성하게 될꺼다.

name: IITP

on:
  push:
    tags: ["*"]
  release:
    types: [created]

jobs:
  build:
    name: Build
    runs-on: ubuntu-18.04
    steps:
      # Git clone
      - name: Checkout
        uses: actions/checkout@v2

      # Set environment variable
      - name: Set environment variable
        run: |
          echo "RELEASE_VERSION=${GITHUB_REF#refs/tags/}" >> $GITHUB_ENV
          echo "REPOSITORY=$(echo $GITHUB_REPOSITORY | tr [:upper:] [:lower:])" >> $GITHUB_ENV

      # Build Front-end
      - name: Setup Node.js
        uses: actions/setup-node@v2
        with:
          node-version: '16'
      - name: Change directory to frontend and build front-end codes using NPM
        run: |
          cd frontend
          npm i
          npm run build

      # Build Back-end
      - name: Setup JDK 1.8
        uses: actions/setup-java@v1
        with:
          java-version: 1.8
          java-package: jdk
      - name: Grant execute permission for gradlew and build back-end codes using Gradle
        run: |
          echo "${{ secrets.DATASOURCE }}" > src/main/resources/datasource.yml
          chmod +x gradlew
          ./gradlew bootWar

      # Build and push Docker image
      - name: Setup Docker Buildx
        uses: docker/setup-buildx-action@v1
      - name: Login to Github Container Registry
        uses: docker/login-action@v1
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.ACCESS_TOKEN }}
      - name: Build and Push
        uses: docker/build-push-action@v2
        with:
          context: .
          push: true
          tags: ghcr.io/${{ env.REPOSITORY }}:release-${{ env.RELEASE_VERSION }}

      # SSH
      - name: Deploy via SSH
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.REMOTE_IITP_HOST }}
          port: ${{ secrets.REMOTE_IITP_SSH_PORT }}
          username: ${{ secrets.REMOTE_IITP_SSH_USERNAME }}
          password: ${{ secrets.REMOTE_IITP_SSH_PASSWORD }}
          script: |
            CONTAINER_NAME=iitp-prd
            TAG=release

            REDEPLOY_LIST=(1 2 3)
            REDEPLOY_NUM=`expr $(($RANDOM% 3)) + 1`

            for IDX in ${REDEPLOY_LIST[@]/$REDEPLOY_NUM}; do
              if [ `docker ps | grep -c $CONTAINER_NAME-$IDX` != 0 ]; then docker stop $CONTAINER_NAME-$IDX; fi
              if [ `docker ps -a | grep -c $CONTAINER_NAME-$IDX` != 0 ]; then docker rm $CONTAINER_NAME-$IDX; fi
            done

            NEW_VERSION_IMAGE=ghcr.io/${{ env.REPOSITORY }}:$TAG-${{ env.RELEASE_VERSION }}

            docker pull $NEW_VERSION_IMAGE
            for IDX in ${REDEPLOY_LIST[@]/$REDEPLOY_NUM}; do
              docker run -d --name $CONTAINER_NAME-$IDX -p `expr 9200 + $IDX`:9200 $NEW_VERSION_IMAGE
            done

            docker stop $CONTAINER_NAME-$REDEPLOY_NUM
            docker rm $CONTAINER_NAME-$REDEPLOY_NUM
            if [ `docker images | grep ghcr.io/${{ env.REPOSITORY }} | grep -v $TAG-${{ env.RELEASE_VERSION }} | grep -c ''` != 0 ]; then
              docker rmi $(docker images -q --filter "dangling=true") $(docker images | grep ghcr.io/${{ env.REPOSITORY }} | grep $TAG | grep -v ${{ env.RELEASE_VERSION }} | awk '{print $3}')
            fi
            docker run -d --name $CONTAINER_NAME-$REDEPLOY_NUM -p `expr 9200 + $REDEPLOY_NUM`:9200 $NEW_VERSION_IMAGE

위 스크립트는 실제로 내가 회사 프로젝트에 적용했던 Github Action의 yml 파일이다.

여기서 작성한 절차를 말로 풀어보면 다음과 같다.

  1. Git 레포지토리 Clone하여 프로젝트를 받아온다.
  2. 환경변수를 지정한다.
  3. 프론트엔드 코드를 빌드한다.
  4. 빌드된 프론트엔드 코드를 백엔드쪽으로 합친 후 애플리케이션 전체를 빌드한다.
  5. 애플리케이션 빌드 파일을 Docker 이미지로 빌드한 후 Github Container Registry에 Push한다.
  6. 배포할 원격지 서버에 SSH로 접속한다.
  7. 배포한다. (쉘 커맨드 설명은 생략)

개 복잡해보여도 하나씩 보자고.

name: IITP

우선, 첫 번째 라인에 name이 있다. 그냥 애플리케이션 이름이다. 자 다음

on:
  push:
    tags: ["*"]
  release:
    types: [created]

Github Action에는 Workflows, Events, Jobs, Steps, Actions, Runners라는 6가지 개념이 존재한다.

  • Events: push, pull, pull request와 같은 이벤트
  • Workflows: 레포지토리에 발생한 이벤트에 대해 트리거되어 이루어지는 자동화된 절차
  • Runners: Github Action의 runner. Workflows가 돌아가기 위한 환경이라 보면 됨.
  • Steps: Workflows 안에서 하나의 일을 수행하는 단일 작업 단위
  • Jobs: 동일한 Runner에서 실행되는 Steps들의 집합
  • Actions: Workflows에서 가장 작은 실행 단위. 외부 action을 갖다 쓸 수 있으며 내가 action을 만들 수도 있음.

Workflows는 쉽게 말하면 저 위의 yml 스크립트 자체가 Workflows다. 그냥 자동화 절차를 나타낸 것이니.

on은 Events에 해당한다.

on이라는 프로퍼티 안에 push와 release라는 프로퍼티가 있다.

on에서는 push, pull, pull request (pr), release 등 트리거 설정을 할 수 있다.

그냥 push를 할 때, 특정 브랜치에 push 할 때, 태깅 후 push 할 때 등을 모두 정할 수 있다.

아래는 Events에 대한 공식 문서다. 더 많은 정보가 있으니 참고하자고.

넘어가자.

jobs:
  build:
    name: Build
    runs-on: ubuntu-18.04
    steps:
			...

jobs라는 큰 단위가 나오고 그 밑에 build, 그 밑에는 name, runs-on, steps가 있다.

name은 그냥 이름이고 (이제부터 name 나오면 설명 안한다. name은 그냥 이름이니까 대충 넘어가자)

runs-on은 이 Workflows가 Runner에서 돌아갈 기반 환경이다.

여기다가는 OS를 적는다.

나는 ubuntu-18.04를 적었다.

steps의 하위부터 본격적으로 내가 자동화 할 절차를 적어가면 된다.

맨 위에 올려놓은 yml 기준으로 설명한다.

  1. Git 레포지토리를 Clone하여 프로젝트를 받아온다.

    git clone을 위해 actions/checkout@v2 라는 Actions가 필요하다.

    uses 프로퍼티에 action을 저렇게 적어주기만 하면 Github Action에서 알아서 외부 action을 끌어와 사용할 수 있다.

    # Git clone
- name: Checkout
  uses: actions/checkout@v2

  2. 환경변수를 지정한다.

    run이라는 프로퍼티에서 커맨드 작성이 가능하다.

    현재 새로 릴리즈 된 버전명과 레포지토리명을 환경 변수로 지정하였다.

    # Set environment variable
- name: Set environment variable
  run: |
    echo "RELEASE_VERSION=${GITHUB_REF#refs/tags/}" >> $GITHUB_ENV
    echo "REPOSITORY=$(echo $GITHUB_REPOSITORY | tr [:upper:] [:lower:])" >> $GITHUB_ENV

  3. 프론트엔드 코드를 빌드한다.

    프론트엔드를 빌드를 하기 위해선 npm이 필요하다.

    npm을 사용하기 위해 actions/setup-node@v2이라는 action을 적어준다.

    그리고 그것의 버전을 적고 싶다면

    with 프로퍼티에 node-version: '버전' 으로 적어주면 된다.

    그리고 run에다가 코드 빌드하는 커맨드를 적어넣는다.

    # Build Front-end
- name: Setup Node.js
  uses: actions/setup-node@v2
  with:
    node-version: '16'
- name: Change directory to frontend and build front-end codes using NPM
  run: |
    cd frontend
    npm i
    npm run build

  4. 애플리케이션 전체를 빌드한다.

    # Build Back-end
- name: Setup JDK 1.8
  uses: actions/setup-java@v1
  with:
    java-version: 1.8
    java-package: jdk
- name: Grant execute permission for gradlew and build back-end codes using Gradle
  run: |
    echo "${{ secrets.DATASOURCE }}" > src/main/resources/datasource.yml
    chmod +x gradlew
    ./gradlew bootWar

    사실 3.에서 빌드하는 과정에서

    mv나 cp같은 커맨드가 아닌

    백엔드(Spring Boot)의 src/main/resources/static 경로에 저장되도록

    빌드 경로를 애초에 package.json에 설정해놨었다.

    그래서 프론트엔드-백엔드가 이미 합쳐진 상태이다.

    Spring Boot로 개발된 애플리케이션이기에

    자바가 필요하다.

    자바에 대한 action은 actions/setup-java@v1 이다.

    그 외 버전이나 패키지같은 건 위에 보이는 대로 적으면 된다.

    나머지는 3.과 마찬가지로 빌드하는 커맨드를 입력한다.

    여기서 ${{ secrets.DATASOURCE }}가 보이는데,

    이건 Github Context라는 거다.

    Github Action에서 쓰는 일종의 변수같은거다.

    보안상의 이유로 application.yml로부터 datasource 정보를

    분리해서 레포지토리에 push 했었는데

    빌드를 하려면 그 정보가 필요했다.

    Github Context는 아래 링크를 통해 확인하자고.

  5. 애플리케이션 빌드 파일을 Docker 이미지로 빌드한 후 Github Container Registry에 Push한다.

    # Build and push Docker image
- name: Setup Docker Buildx
  uses: docker/setup-buildx-action@v1
- name: Login to Github Container Registry
  uses: docker/login-action@v1
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.ACCESS_TOKEN }}
- name: Build and Push
  uses: docker/build-push-action@v2
  with:
    context: .
    push: true
    tags: ghcr.io/${{ env.REPOSITORY }}:release-${{ env.RELEASE_VERSION }}

    Github Container Registry란 Github에서의 도커 이미지 저장소다.

    도커 이미지를 만들기 위한 action으로 docker/setup-buildx-action@v1가 필요하고,

    도커 이미지를 push하기 위해 docker/build-push-action@v2가 필요하다.

    push하려면 도커 레지스트리에 로그인해야하므로 docker/login-action@v1 또한 필요하다.

    도커 로그인에서의 with 프로퍼티를 보면 로그인 할 registry, 아이디, 비번을 적어야 한다.

    Github Container Registry은 레지스트리 이름이 ghcr.io이다.

    만약 Docker hub나 다른 레지스트리를 이용중이라면 그에 맞게 적어넣으면 된다.

    도커 이미지 빌드 작업에서 context, push, tags가 보이는데,

    context는 Dockerfile이 위치한 경로다.

    Workflows에서는 기본적으로 프로젝트 루트 디렉토리가 현재 디렉토리다.

    난 처음부터 Dockerfile을 루트 디렉토리에 놨기 때문에 '.' 만 적었다.

    push에는 boolean 값을 넣는다. 그냥 true로 넣어라.

    그리고 tags에는 레지스트리에 올릴 이미지 이름을 적으면 된다.

  6. 배포할 원격지 서버에 SSH로 접속한다.

    # SSH
- name: Deploy via SSH
  uses: appleboy/ssh-action@master
  with:
    host: ${{ secrets.REMOTE_IITP_HOST }}
    port: ${{ secrets.REMOTE_IITP_SSH_PORT }}
    username: ${{ secrets.REMOTE_IITP_SSH_USERNAME }}
    password: ${{ secrets.REMOTE_IITP_SSH_PASSWORD }}
    script: ...

    Github Action에서 SSH를 다루기 위해선

    appleboy/ssh-action@master라는 action이 필요하다.

    그리고 with에다가 접속할 서버의 IP, 포트, 계정정보를 적어주고

    script에 배포 스크립트를 적으면 된다.

지금까지 설명한 내용은 내가 적용했던 애플리케이션에 대한

Github Action의 Workflows일 뿐이다.

만약 백엔드가 ExpressJS나 Next.js 같은 Node.js 기반일 경우

Node.js에 대한 action만으로 빌드할 수도 있고,

C/C++로 작성한 프로그램이라면

gcc나 MinGW와 같은 컴파일러에 대한 action을 사용할 수도 있다.

그만큼 자유도가 높고 사용하기도 편해서

앞으로 프로젝트 할 때마다 자주 애용하려 한다.

참고로, Actions를 뭘 써야할 지 모르겠다면

아래 링크로 들어가면 마켓 플레이스가 나온다.

내가 원하는 키워드에 대해 검색하면 관련 action들이 많이 나온다.
