사용자 경험 개선: 멘붕 유저 감지하는 법

2025-04-01

시작 전에

클릭 연타하고 화내는 유저는 다 이유가 있습니다

요즘 사용자는 마음이 여립니다.

스크롤 몇 번 하다가 원하는 정보가 안 보이면 바로 앱 삭제 버튼을 누릅니다.

버튼이 한 번 안 눌리면 앱에 악플을 남기고 떠나는 경우도 많습니다.

UX에서 삐끗하는 순간, 리뷰에는 "별 하나도 아까움" 같은 평가가 올라옵니다.

Datadog의 RUM(Real User Monitoring)은 이런 사용자들의 짜증과 멘붕을 Frustration Signals라는 이름으로 수집해줍니다.

우리는 한국인이니까 좌절 신호라고 부르겠습니다.

사용자가 분노를 표현했는지 아닌지를 로그로 남겨주며

그 시점과 위치를 보면 유추할 수 있습니다.

이 시점에서 멘탈이 흔들렸구나 하는 식으로요.

이런 로그가 쌓이면? 바로 우리 팀 회의실에서 사운드 이펙트 나오는 겁니다.

좌절 신호의 종류

Datadog은 대표적으로 세 가지 좌절 신호를 추적합니다.

사용자가 직접 말하지 않더라도 손가락에 모든 감정이 담긴 순간들입니다.

개발자한테 직접 욕은 못 하지만 클릭으로는 충분히 표현할 수 있거든요.

1. Dead Clicks (죽은 클릭)

클릭했는데 아무런 반응이 없습니다.

겉보기에는 버튼인데 눌러도 무반응입니다.

가끔 버튼처럼 생겼는데 div에 onClick도 없는 구라 버튼인 경우가 있습니다.

이럴 땐 버튼이 안눌리니 발작 버튼이라도 누르게 됩니다.

사용자는 반복해서 클릭하다가 결국 떠납니다.

예전 프로젝트에서 디자이너가 시멘틱 마크업을 몰랐던 시절 떠올리시면 됩니다.

겉보기만 번지르르한 인터랙션은 기만일 뿐입니다.

눈으로 낚고 기능으로 배신하는 UI는 그냥 UX 사기입니다.

2. Rage Clicks (분노 클릭)

한 지점을 초당 3번 이상 클릭하는 것은 지금 사용자가 격앙돼 있다는 뜻입니다.

기능이 작동하지 않거나 피드백이 부족할 때 자주 발생합니다.

이는 UI 또는 UX 설계 자체에 구조적인 문제가 있다는 신호입니다.

이걸 UX 실수라고 넘기시면 안 됩니다.

문제는 반복해서 겪은 사용자가 서비스에 대해 부정적인 입소문을 낸다는 점입니다.

서비스는 안 커도 욕은 잘 퍼집니다.

분노 클릭은 유저가 마지막 남은 인내심을 집어던지는 순간임을 명심해야 합니다.

3. Error Clicks (에러 클릭)

버튼을 클릭했는데 콘솔이나 브라우저에 에러가 터지는 경우입니다.

이건 말 그대로 시스템이 사용자 요청을 처리 못하고 그냥 터졌다는 뜻입니다.

사용자는 무슨 일이 일어났는지도 모른 채 멈춰있는 화면을 보며 혼란스러워하고

개발자는 이 로그를 보고 가슴이 철렁할 수도 있습니다.

보통 무심코 지나가는 JS 오류, 백엔드 응답 실패, 존재하지 않는 DOM 조작 시도 등이 원인입니다.

즉, 코드가 유저 경험에 제대로 뒷통수를 날리는 순간입니다.

이걸 왜 봐야 하느냐면

슬슬 쪽팔리니까 Frustration Signals라고 하겠습니다.

Frustration Signals는 단순한 통계 데이터가 아닙니다.

사용자가 불편함을 느끼고 있다는 사실을 조용히 전달하는 방식입니다.

Datadog은 이 흔적들을 모아서 로그로 보여줍니다.

분노 클릭이 집중된 페이지나, 죽은 클릭이 반복된 요소를 보면

누가, 언제, 어디서 불편함을 느꼈는지를 알 수 있습니다.

UX 회의에서 사용자 경험 개선이 필요하다는 말을 하기가 애매하신 경우

Frustration 로그 한 장만 첨부하시면 됩니다.

"이 화면에서 분노 클릭 84건, 에러 클릭 27회 발생"

이런 숫자 앞에서는 누구도 반박하기 어렵습니다.

숫자는 거짓말을 안 합니다. 다만 기획서를 곤란하게 만들 뿐입니다.

결론

Frustration Signals는

사용자가 ‘지금 뭔가 불편하다’고 조용히 알리는 로그입니다.

이 신호를 무시하면 사용자는 조용히 떠나고

서비스는 별점 테러를 맞으며

개발팀은 소방수 모드로 전환됩니다.

Datadog RUM은 Frustration Signals를 자동으로 기록해줍니다.

따라서 우리가 해야 할 일은, 이 로그를 보고 상황을 개선하는 것입니다.

수치 기반으로 사용자 고통 포인트를 가시화하고

그걸 바탕으로 UI/UX를 개선해 나가면 됩니다.

사용자가 짜증 낼 시간에 우리는 먼저 로그를 보고 고치고

그걸 본 기획자도 안심하고

서비스 평점도 올라가게 됩니다.

그리고 다음 날 또 분노 클릭 로그 300건이 떠 있으면

그건 다시 회의 소집하시면 됩니다. 점심은 미리 드시고요.
