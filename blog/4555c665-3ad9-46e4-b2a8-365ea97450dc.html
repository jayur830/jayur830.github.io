File Convention

default.(jsx?|tsx)

Parallel Route에서 하위 path 접근 시 특정 세그먼트에 해당 path가

존재하지 않을 때 기본으로 보여줄 컴포넌트 (이게 뭔말이야)

예를 들어, 위 이미지처럼 @team, @analytics라는 세그먼트가 존재하고

@team의 하위에만 settings라는 path의 페이지가 존재할 때,

브라우저에서 /settings로 진입 시 @team에 해당하는 부분은 제대로 보여주지만

@analytics에 해당하는 부분은 보여줄 것이 없다.

이 때 default.(jsx?|tsx) 파일이 하위에 저렇게 존재할 경우

그걸 보여주게 된다.

page.(jsx?|tsx)

특정 페이지

layout.(jsx?|tsx)

page.(jsx?|tsx)의 껍데기에 해당하는 부분

template.(jsx?|tsx)

layout.(jsx?|tsx)과 비슷한 껍데기

하지만 서로 다른 하위 path 라우팅 시 상태를 잃어버린다.

(예: useState로 선언된 상태 또는 useEffect로 실행될 로직이 매 라우팅 시 새로 실행)

error.(jsx?|tsx)

404 이외에 특정 페이지 내에서 에러가 발생했을 때 보여줄 페이지

app 디렉토리 내 루트 또는 각 페이지 디렉토리 내에 위치하며,

'use client';를 최상단에 선언해야 한다. (클라이언트 컴포넌트 전용)

loading.(jsx?|tsx)

클라이언트에 페이지를 띄우기까지의 시간동안 보여줄 컴포넌트

not-found.(jsx?|tsx)

404 페이지

app 디렉토리 내 루트에 위치해야 한다.

이 각각의 File Convention들은 내부적으로 다음을 수행한다. (그림 진짜 쵝오)

Routing

(name)

Group Routes를 위한 컨벤션으로,

적용 시 path의 (name) 부분이 생략된다.

(예: 디렉토리 경로가 app/(name)/a/b/c 일 경우 실제 url path는 /a/b/c)

(..)name

상위 path를 생략.

(예: 디렉토리 경로가 app/a/(..)b/c 일 경우 실제 url path는 /b/c)

그리고 만약 /b/c가 app의 루트 레벨에 똑같이 존재할 경우

라우팅 시에는 app/a/(..)b/c 를, 새로고침 시에는 app 루트 레벨의 /b/c를 바라보게 된다.

(..)(..)name

(..)name과 비슷하다. (..)가 두번 쓰였으니 상위의 상위 path까지 생략.

(...)name

url path 루트 레벨까지 생략 후 /name 으로 url 적용.

@name

Parallel Route를 위한 컨벤션으로, 이것은 url path가 아니다.

각 page의 props로 name이라는 ReactNode 타입의 props가 전달된다.

Route Handler (구 API Route)

route.\[jt\]s

기존 pages/api 디렉토리에서 Next.js API 핸들러 역할을 하던 놈과 똑같음.

단, page.(jsx?|tsx)와 같은 위치에 존재하면 안된다.

url path가 같아서 충돌나기 때문.

만약 app/foo 디렉토리에 page.tsx와 route.ts가 있을 때

/foo 라는 path는 이미 foo 페이지의 url path인데

동시에 /foo 엔드포인트를 가진 api이기도 하기 때문에 충돌이 난다.
