Nest.js는 Node.js 기반의 백엔드 프레임워크이다.

Express와 Fastify 중 하나를 래핑하여 동작하는 데, 기본으로 Express로 동작한다.

Install

설치는 다음 커맨트를 입력하면 Nest.js 보일러플레이트가 생긴다.

npm i -g @nestjs/cli
nest new <project-name>

Module

모듈이란 위의 그림처럼 Nest.js 애플리케이션을 구성하는 단위이다.

Nest.js으로 개발된 모든 애플리케이션에는 최소 하나의 루트 모듈이 있으며,

이 루트 모듈부터 트리처럼 다른 모듈들을 포함한다.

// src/cats/cats.module.ts

import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {}

// src/dogs/dogs.module.ts

import { Module } from '@nestjs/common';
import { DogsController } from './dogs.controller';
import { DogsService } from './dogs.service';

@Module({
  controllers: [DogsController],
  providers: [DogsService],
})
export class DogsModule {}

// src/app.module.ts

import { CatsModule } from './cats/cats.module';
import { DogsModule } from './dogs/dogs.module';

@Module({
  imports: [CatsModule, DogsModule],
})
export class AppModule {}

이 @Module 데코레이터에는 4개의 속성을 넣을 수 있다.

  • imports: 모듈에 포함할 다른 모듈의 집합 (Array)
  • exports: 다른 모듈에서 사용 가능하도록 내보낼 Provider(Service)의 집합 (Array)
  • providers: 모듈에 주입될 Provider의 집합 (Array)
  • controllers: 모듈에 주입될 Controller의 집합 (Array)

Provider와 Controller는 이 뒤에 설명한다.

Controller

Controller는 클라이언트의 request를 받아 response를 응답해주는 부분이다.

그리고 코드는 요렇게 생겼다.

import { Body, Controller, Delete, Get, Param, Post, Put } from '@nestjs/common';

import { UserVO } from '@/vo/user.vo';

@Controller('foo')
export class FooController {
  @Get('/find')
  find(): string {
    return 'get';
  }

  @Post('/insert')
  insert(@Param() name: string): void {
    console.log('insert name:', name);
  }

  @Put('/update')
  update(@Body() user: UserVO): void {
    console.log('update user:', user);
  }

  @Delete('/remove')
  remove(@Param() id: string): string {
    console.log('delete id:', id);
    return 'delete';
  }
}

각각 HTTP 메소드인 GET, POST, PUT, DELETE 요청(각 메소드 매개변수)을 받아서 처리한 후 응답(return)한다.

근데 이거 어디서 많이 보지 않았나..?

package com.app.controllers.foo;

import com.app.vo.UserVO;

import lombok.extern.slf4j.Slf4j;

import org.springframework.web.bind.annotation.DeleteMapping;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.PutMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@Slf4j
@RestController()
public class FooController {
  @GetMapping("/find")
  public String find() {
    return "get";
  }

  @PostMapping("/insert")
  public void insert(@RequestParam() final String name) {
    this.log("insert name: ", name));
  }

  @PutMapping("/update")
  public void update(@RequestBody() final UserVO user) {
    this.log("update user: ", user);
  }

  @DeleteMapping("/remove")
  public String remove(@RequestParam() final String id) {
    this.log("delete id: ", id);
    return "delete";
  }
}

문법적으로 뭔가 Spring이랑 많이 비슷한거 같지만

Spring은 Java 언어이고 멀티스레드 기반이다.

Nest.js는 Node.js 기반의 싱글스레드이기 때문이 매커니즘은 완전히 다르다.

(사실 Python의 Flask도 문법적으로는 비슷하다 :0)

아무튼 Controller가 이런거고 흠흠

근데 이 Controller에서 요청에 대해 처리를 하진 않는다.

실제로 처리를 담당하는 비즈니스 로직은 Provider라는 친구가 하게 되고

Controller는 그걸 받아서 그대로 클라이언트에 넘겨줄 뿐이다.

Provider

Provider는 앞서 말한대로 비즈니스 로직을 처리하는 친구이다.

요놈은 Service가 될 수도 있고 Repository, Factory, Helper라는 애가 될 수도 있다고 한다.

(사실 이 글을 쓸 때 Factory, Helper는 처음 알았다)

그리고 의존성 주입(Dependency Injection, 이하 DI)이라는 개념이 등장한다.

(너무 Spring이랑 비슷해지는데)

@Injectable이라는 데코레이터가 붙은 클래스는 Provider가 된다.

이 Provider들은 싱글톤으로 존재하여, 다른 Provider나 Controller로 주입될 수 있다.

뭔 소리냐면

import { Injectable } from '@nestjs/common';
import { Cat } from './interfaces/cat.interface';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  create(cat: Cat) {
    this.cats.push(cat);
  }

  findAll(): Cat[] {
    return this.cats;
  }
}

요런 냥냥이 Provider가 있다고 하고

// src/cats/cats.module.ts

import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {}

냥냥이 Module이 있을 때

import { Controller, Get, Post, Body } from '@nestjs/common';
import { CreateCatDto } from './dto/create-cat.dto';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

@Controller('cats')
export class CatsController {
  constructor(private catsService: CatsService) {}

  @Post()
  async create(@Body() createCatDto: CreateCatDto) {
    this.catsService.create(createCatDto);
  }

  @Get()
  async findAll(): Promise<Cat[]> {
    return this.catsService.findAll();
  }
}

이런 식으로 Controller에 주입할 Provider를 생성자의 매개변수로 작성만 하면

해당 Controller에서는 이 Provider를 사용할 수 있다.

이것 말고도 @Inject 데코레이터를 클래스 내 private 멤버 변수에 붙혀서 주입하는 방법도 있다.

(Spring의 @Autowired 처럼)

Spring과 Nest.js를 모두 써본 입장으로써

솔직히 아직은 Spring이 더 편한 것 같다.

이 글을 쓰고 TypeORM에 대해 또 쓸 예정인데

TypeORM은 트랜잭션을 비즈니스 로직 앞뒤에 감싸야한다.

Spring은 @Transactional 어노테이션만 붙히면 끝이다.

이 밖에도 Nest.js에서는 DI가 될려면 Provider를 일단 모듈에도 선언해야 하는데

Spring은 그런거 없이 바로 주입 가능한 점 등등

아직 Nest.js가 익숙하지 않고 잘 몰라서 그러는거 일꺼라고 믿어야겠다.
