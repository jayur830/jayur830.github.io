Intercepting Routes

이해하는데에는 2일이 걸렸지만 이해하기 귀찮아서 몇 달동안 그냥 모르고 살다가 이제서야 공부했습니다.

Convention

이렇다고 합니다.

  • (.) : 현재 세그먼트의 path 매칭
  • (..) : 부모 세그먼트의 path 매칭
  • (..)(..) : 두 단계 세그먼트 위의 path로 매칭
  • (...) : 루트에 존재하는 path로 매칭

세그먼트는 쉽게 말해 app 디렉토리 하위의 각 페이지 디렉토리 레벨이라고 생각하시면 편합니다.

(Docs에는 디렉토리 기준이 아니라고는 하는데 뭐 아무튼)

혹시 몰라서 (..)(..)(..) 나 (..)(..)(..)(..)(..)(..)(..) 이런 식의 도배도 해봤는데

이건 안먹히더군요

Example

다음과 같은 app 디렉토리 구조가 있다고 가정합시다.

/app
  ├── a
  │    ├── b
  │    │    ├── x
  │    │    │    ├── (..)(..)c
  │    │    │    │     └── page.tsx
  │    │    │    ├── (..)y
  │    │    │    │     └── page.tsx
  │    │    │    └── page.tsx
  │    │    └── y
  │    │         └── page.tsx
  │    ├── c
  │         └── page.tsx

그리고 각 페이지별로 이렇게 작성되어 있습니다.

// app/a/b/x/page.tsx

import Link from 'next/link';

export default function Page() {
  return (
    <>
	    <div>
		    <Link href="/a/c">Go /a/c</Link>
		  </div>
	    <div>
		    <Link href="/a/b/y">Go /a/b/y</Link>
		  </div>
    </>
  );
}

// app/a/b/x/(..)(..)c/page.tsx

export default function Page() {
  return '/ a / b / x / (..)(..)c';
}

// app/a/b/x/(..)y/page.tsx

export default function Page() {
  return '/ a / b / x / (..)y';
}

// app/a/c/page.tsx

export default function Page() {
  return '/ a / c';
}

// app/a/b/y/page.tsx

export default function Page() {
  return '/ a / b / y';
}

그리고 브라우저 주소 창에 /a/b/x 를 입력하고 엔터를 눌러 해당 주소로 진입합니다.

그리고 먼저 Go /a/c 로 되어있는 링크를 클릭해보세요.

/ a / b / x / (..)(..)c 가 페이지에 노출됩니다.

그 상태에서 새로고침을 하면 / a / c 가 새로 노출됩니다.

신기하지 않습니까

마찬가지로 다시 /a/b/x 주소로 돌아와서 Go /a/b/y 링크를 클릭해봅시다.

이번엔 / a / b / x / (..)y 텍스트가 페이지에 보여집니다.

여기서도 새로고침을 하면 페이지의 컨텐츠는 / a / b / y 라는 텍스트로 다시 바뀌게 됩니다.

Intercepting Routes라는 단어 그대로 해석하여 라우팅 경로를 가로채서

원래 페이지로 라우팅하는 것이 아닌 도둑(?) 라우팅을 하는 개념으로 이해하면 되겠습니다

주로 Parallel Routes와 함께 사용하여 모달을 구현하는데 사용된다고 합니다

참고로, 이 기능이 제대로 동작하기 위해서는 (.) , (..) , (..)(..) 등의 기호가 붙은 디렉토리와 동일 레벨에 page.tsx 파일이 있어야 하며, 해당 page.tsx 에 Intercepting Routes로 향하는 Link 가 있어야 합니다. 이거 알아낼려고 하루 버림
