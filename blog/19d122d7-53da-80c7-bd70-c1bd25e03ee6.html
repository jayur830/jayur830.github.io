웹 & 앱 성능 최적화 및 회고

2025-02-17

문제 상황

최근 사내 어떤 서비스에서 성능 이슈를 경험했습니다. 이 사태로 인해 일주일 간 고객 항의 전화가 빗발치고 스토어 별점 테러까지 맞았구요

특히, 앱의 메모리 사용량 증가 및 웹의 로딩 지연 문제가 발생했으며, 주요 증상은 다음과 같았습니다.

  • 저사양 폰에서 페이지 로드가 원활하지 않음 → 흰 화면이 지속적으로 노출
  • 웹뷰 캐시가 적용되지 않음 → HTML, CSS, JS, 이미지 등의 리소스를 매번 네트워크에서 다운로드하여 불필요한 데이터 사용 증가 (과금 위험 발생)

이에 대한 원인을 분석하고 최적화 작업을 진행했습니다.

원인 분석

  1. 앱 메모리 사용량 증가

    • 앱 내부에 불필요한 리소스가 다량 포함되어 있었습니다.

    • 불필요한 코드, 라이브러리가 메모리를 과다 사용하면서 성능 저하가 발생했습니다.

  2. 웹 성능 저하

    • 사용하지 않는 이미지와 코드가 포함되어 있어 웹 번들 크기가 커졌습니다.
    • 캐시가 적용되지 않아 동일한 리소스를 반복적으로 다운로드하는 문제가 발생했습니다.

해결

  1. 폰트 최적화
    • 기존 폰트 파일을 WOFF 형식으로 변경하여 파일 크기를 줄이고 로딩 속도를 개선했습니다.
  2. 이미지 최적화
    • PNG 형식의 이미지를 WebP 형식으로 변환 후 사용하여 LCP를 개선했습니다.
  3. 불필요한 리소스 정리
    • 사용하지 않는 이미지, 코드, 라이브러리를 삭제했습니다.
    • CSS, JS 코드에서 트리 쉐이킹(Tree Shaking) 적용하여 불필요한 코드를 제거했습니다.
  4. 번들러 설정 최적화

    • 앱 빌드 시 불필요한 모듈을 제외하여 번들 크기를 줄였습니다.

      // metro.config.js

...

module.exports = {
  ...
  resolver: {
    ...
    /** @description 웹 빌드 시 제외하지 않음 */
    blacklistRE:
      process.env.PLATFORM === 'web'
        ? undefined
        : exclusionList([
            /@\/assets\/.*/, // assets 디렉토리 제외
            /\(auth\)\/.*/, // (auth) 디렉토리 제외
            /\(main\)\/.*/, // (main) 디렉토리 제외
            /\(popup\)\/.*/, // (popup) 디렉토리 제외
            /.*\.web\.tsx$/, // .web.tsx 파일 제외
          ]),
    ...
  },
};

    • 웹에서도 코드 스플리팅(Code Splitting) 적용하여 필요한 리소스만 로드하도록 개선했습니다.

  5. 캐싱 정책 적용
    • 웹 서버로 NGINX를 사용하고, 모든 리소스에 대해 5분(max-age=300) Cache-Control 을 적용했습니다.
    • 이를 통해 불필요한 네트워크 요청을 줄이고, 빠른 로딩 속도를 유도했습니다.

결과

APP

  • AS-IS
    • Java: 약 15MB
    • Native: 165.1MB
    • Graphics: 약 129MB
    • Stack: 76KB
    • Code: 134.1MB
    • Other: 71.1MB
  • TO-BE
    • Java: 약 22.5MB (50% 증가)
    • Native: 36.2MB (78% 감소)
    • Graphics: 약 110.4MB (14.4% 감소)
    • Stack: 44KB (42% 감소)
    • Code: 58MB (56.7% 감소)
    • Other: 58.5MB (17.7% 감소)

WEB

  • AS-IS
    • 빌드 후 전체 파일 용량: 157MB
    • 페이지 로드 시간: 1.35초
    • 번들 크기: 28.9MB
    • 폰트 파일 평균 크기/개: 1.4MB
    • LCP: 2.05
  • TO-BE
    • 빌드 후 전체 파일 용량: 123MB
    • 페이지 로드 시간: 1.22초
    • 번들 크기: 1.8MB
    • 폰트 파일 평균 크기/개: 1.1MB
    • LCP: 1.48

SUMMARY

  • 불필요한 라이브러리 삭제 후 Native 메모리 점유율 78% 감소
  • 앱에서 사용하지 않는 코드 제외 후 빌드, Code 메모리 점유율 56.7% 감소
  • 이미지 WebP 변환 후 LCP 28% 감소
  • 폰트 파일 형식 woff로 변경, 네트워크 폰트 응답 용량 21% 감소
  • 사용하지 않는 Asset 삭제하여 전체 빌드 파일 용량 21.7% 감소
  • Code Splitting을 통해 Bundle Size 94% 감소

마무리

  1. 배포 전 테스트의 중요성
    • 사실 당연한건데, 배포 전에 다양한 환경에서 철저하게 테스트하는 것이 필수임을 다시금 깨닫게 되었습니다. 단순히 개발 환경에서 잘 동작하는 것만으로는 충분하지 않았습니다.
  2. 캐싱 전략의 중요성
    • 적절한 캐싱 정책을 적용하는 것만으로도 성능이 크게 향상될 수 있었습니다. 하지만 너무 길거나 짧은 캐싱 시간은 오히려 역효과를 낼 수 있기 때문에 상황에 맞는 설정이 중요할 것 같습니다.
  3. 최적화는 지속적인 과정
    • 한 번의 최적화로 끝나는 것이 아니라, 계속해서 모니터링하고 개선해 나가는 과정이 필요합니다.
  4. 오피셜 네임드 프레임워크 100% 맹신 금지
    • 프레임워크나 라이브러리 등 채신기술 도입할 때에는 그게 유명하든 말든 일단 검증은 한번 제대로 하고 시작해야할 것 같습니다.

이번 성능 최적화 과정을 통해 불필요한 리소스를 줄이고, 캐싱을 통해 최적화하는 것만으로도 앱과 웹의 성능이 크게 개선될 수 있음을 확인했습니다.

일정을 맞추는 것에 집중하다 보니 정확성을 놓쳤고, 결국 문제가 발생했습니다. 빠르게 끝내는 것보다 정확하게 수행하는 것이 더 중요하다는 것을 깨달았습니다.
