이딴 걸 왜 써야 하나요

리액트 프로젝트 하다 보면 전역 상태 관리가 필요하다면서 Redux, Zustand, Recoil 같은 걸 쓰라고 선동하는 나쁜 사람들이 있는데

특히 API에서 받아온 데이터를 전역 상태로 관리해야 한다는 얘기를 들으면 조금 어이가 없습니다

API에서 받아오는 데이터는 서버 상태(server state) 입니다

전역 상태(global state)와는 아예 다릅니다

  • 전역 상태: UI 열림 여부, 사용자 입력 값, 로그인 정보 등
  • 서버 상태: API에서 받아오는 데이터 (예: 사용자 목록, 게시글 리스트, 상품 정보 등)

서버 상태를 전역 상태 관리 도구에 넣으면 뭐가 문제냐면

  • API 데이터 갱신될 때마다 상태도 직접 관리해야 함
  • 불필요한 리렌더링 발생해서 성능 손해 봄
  • 캐싱, 동기화 같은 것도 전부 직접 처리해야 함

React Query, SWR, Apollo Client 같은 거 쓰면 이런 문제 전부 해결됩니다

해결책: React Query, SWR, Apollo Client 쓰면 됩니다

이런 라이브러리는 이미 잘 만들어져 있어서

우리가 힘 빼면서 직접 API 데이터 관리할 필요가 없습니다

  • React Query, SWR → REST API 데이터 관리 끝판왕. 자동 캐싱, 리페치, 상태 동기화 기능 다 있음
  • Apollo Client → GraphQL 쓸 때 필수. 캐싱, 데이터 관리, 상태 동기화 깔끔하게 해줌

예제 코드 (React Query)

const { data, error, isLoading } = useQuery(['todos'], fetchTodos)

이거 한 줄이면 데이터 가져오고, 로딩 상태 관리하고, 캐싱까지 자동으로 해줍니다

Redux, Zustand 같은 거 쓰면 useEffect, dispatch, reducer, selector 이런 거 다 만들어야 하는데

이게 훨씬 간단하지 않나요

전역 상태 관리가 필요한 경우

그럼 전역 상태 관리 도구는 언제 써야 할까요

API 데이터 말고 이런 거 관리할 때 씁니다

  • 사용자 인증 정보 (로그인 여부, JWT 토큰 등)
  • UI 상태 (사이드바 열림 여부, 다크 모드 설정 등)
  • 웹소켓 실시간 데이터 (채팅 메시지, 알림 등)

이런 건 서버 상태와 다르게 자주 바뀌고

전역으로 공유해야 하기 때문에 Zustand, Jotai, Recoil 같은 걸 쓰는 게 유용합니다

결론: API 데이터는 React Query 쓰면 됩니다

API 데이터는 전역 상태 관리할 필요가 없습니다

React Query, SWR, Apollo Client 같은 서버 상태 관리 도구를 쓰면 알아서 다 해줍니다

Redux, Zustand, Recoil 같은 건 꼭 필요한 곳에서만 써야 합니다

괜히 쓸데없이 복잡하게 만들지 말고 코드 단순하게 가는 게 중요합니다

그게 개발자의 행복을 지키는 길입니다
